dominion-observatory-sdk

Cross-ecosystem agent behavioral telemetry. The only MCP scoring network that accepts agent-reported runtime data. MIT no install zero deps

Breaking change in 0.2.0 — agent_id is required

Every report() and instrument() call must include a stable agent_id string. Missing, empty, or reserved values (anonymous, observatory_probe) throw synchronously. A package name or a persisted UUID both work. This is how the Observatory attributes cross-ecosystem demand to real external agents instead of lumping everything under one anonymous bucket.

TypeScript / JavaScript / Deno / Bun

import { report, checkTrust, instrument } from "https://sdk-cdn.sgdata.workers.dev/v1/observatory.mjs";

const AGENT_ID = "acme-scheduler@1.2.0"; // package name or persisted UUID

// Fire-and-forget telemetry
await report({
  agent_id: AGENT_ID,
  server_url: "https://my-mcp-server.example.com/mcp",
  success: true,
  latency_ms: 142,
  tool_name: "get_holidays",
});

// Read a trust score
const score = await checkTrust("https://some-other-mcp.example.com/mcp");
if (score.found && (score.trust_score ?? 0) >= 70) await callThatServer();

What gets sent

Exactly six fields per report: agent_id, server_url, success, latency_ms, tool_name, http_status. No query content, no user data, no IPs. The agent_id is a caller-chosen stable identifier — the only field the Observatory uses for cross-ecosystem attribution. Compatible with Singapore PDPA, IMDA Model AI Governance Framework, and EU AI Act Article 12 logging.

Why agent-reported runtime?

Every other MCP scoring platform (Glama, Smithery, MCP Scorecard, Nerq, Zarq) scores servers from the outside using GitHub and registry metadata. Dominion Observatory is the only network that observes what happens when an agent actually calls a tool. Runtime data cannot be backfilled — it compounds temporally and is the strongest data moat in the ecosystem.
